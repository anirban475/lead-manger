## 🔍 Feasibility Analysis of "Update 3" Decisions

### 1. Multi-number phone parsing and integration
* **Feasibility:** **High**.
* **Gotcha (Next.js Hydration Mismatch):** When implementing the Smart WhatsApp Link (which routes mobile to `wa.me` and desktop to `web.whatsapp.com`), checking the user agent or `window` object directly in the render path will cause Next.js to throw a Hydration Mismatch error. This happens because the HTML generated by the server won't match the client-side render.
* **Fix:** Use a standard `isMounted` hook or a `useEffect` block in `CallButtons.tsx` to handle client-side rendering safely:
  ```typescript
  const [isDesktop, setIsDesktop] = useState(false);
  useEffect(() => {
    setIsDesktop(!/Android|iPhone|iPad|iPod/i.test(navigator.userAgent));
  }, []);
  ```

### 2. Nginx Login Rate Limiting (Server Actions)
* **Feasibility:** **Medium-High**.
* **Gotcha:** Next.js uses the same page route (`POST /login`) to submit the Server Action for logging in.
* **Recommendation:** Apply the Nginx `limit_req` block specifically to `POST` requests targeted at `/login` or `/api/auth` paths. Since `/login` only serves the auth form, rate-limiting all `POST` requests to this path is safe and will not interfere with the authenticated application pages:
  ```nginx
  limit_req_zone $binary_remote_addr zone=login_limit:10m rate=1r/s;

  location = /login {
      limit_req zone=login_limit burst=5 nodelay;
      proxy_pass http://127.0.0.1:3020;
      ...
  }
  ```
  *(Define this in your virtual host at `/etc/nginx/sites-available/leads.amatec.in`)*

### 3. Unified Activity Timeline (Logs + Comments)
* **Feasibility:** **High**.
* **Recommendation:** Instead of executing a SQL `UNION` query (which complicates TypeScript type signatures due to mismatched columns), fetch `telecall_logs` and `lead_comments` via two separate database queries in `page.tsx` and merge/sort them in memory using JavaScript:
  ```typescript
  const calls = await getLeadCalls(key);
  const comments = await getLeadComments(key);
  const timeline = [
    ...calls.map(c => ({ type: 'call', date: c.called_at, data: c })),
    ...comments.map(cm => ({ type: 'comment', date: cm.created_at, data: cm }))
  ].sort((a, b) => new Date(b.date).getTime() - new Date(a.date).getTime());
  ```
  This is cleaner, type-safe, and highly performant for individual lead views.

## 🛡️ Database Grants Feasibility
The SQL syntax for column-level updates is correct and supported natively by Postgres:
```sql
GRANT SELECT ON leads TO telecaller_app;
GRANT UPDATE (status, next_action, next_action_date,
              last_disposition, last_called_at, call_count) ON leads TO telecaller_app;
```
This is highly recommended to safeguard critical scoring and scraper metrics.
